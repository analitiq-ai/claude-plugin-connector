# Endpoint identity

How `endpoint_id` relates to the resource it addresses — one model across API
and database connectors.

## The rule

`endpoint_id` is a **lookup handle, not the target.** The engine uses it only to
resolve the on-disk file (`endpoints/{endpoint_id}.json`) and to namespace
checkpoint / state; it never parses the id to recover a table, schema, or URL.
The **verbatim locator** lives in a separate field the engine reads:

- **API** — `operations.<op>.request.path`; the request URL is built from this.
- **Database** — `database_object.{catalog,schema,name}`, stored exactly as
  introspected (case and special characters preserved); the engine dialect-quotes
  it into the qualified SQL identifier. (The engine resolves on `schema` + `name`
  today; three-part `catalog` qualification is a pending engine capability.)

Because the engine acts on the verbatim field, the handle can be derived,
sanitized, or hashed freely — as long as it is unique within its owner and
matches `^[a-z0-9][a-z0-9_-]*$`.

> **`endpoint_id` = a deterministic, collision-safe handle derived from the
> resource's full locator; the verbatim locator is always *also* stored in the
> field the engine reads.**

Same locator ⇒ same handle on every re-author / re-discovery (idempotent).

## Derivation differs by source material, not philosophy

- **API — lossless flatten.** URL paths are charset-safe, so the full locator
  encodes losslessly into the handle. The exact operational rule (delimiter,
  version handling, path-param treatment, worked example) lives in the
  `resources[].key` description in `io-contracts.md` — authoritative there, not
  restated here. The contract enforces the derivation: the
  `endpoint-id-locator` check recomputes the handle from the read operation's
  `request.path` and errors when the authored id diverges.
- **Database — sanitizing slug + hash.** Object names are hostile to the
  charset (case-sensitive quoted identifiers, dots, spaces, unicode), so a
  lossless encode is impossible. The handle is
  `slug(schema)__slug(table)[__slug(catalog)]__<hash8>`, where `<hash8>` is the
  first 8 hex of a stable hash over the *exact verbatim* `catalog.schema.name`:
  `Sales."Order Items"` → `sales__order_items__0e62f7e9`. The slug keeps it
  legible; the hash guarantees uniqueness and determinism.

  The database derivation ships in the contract package as
  `analitiq.contracts.endpoint_identity.derive_db_endpoint_id`, so it is never
  reimplemented — this plugin authors no database endpoints, and the runtime
  discovery that does produce them derives the id from that function. (The API
  flatten lives in the validator, which recomputes it to enforce
  `endpoint-id-locator`.)

## Why "encodes the locator" (API) and "opaque handle" (DB) are the same rule

Even the API handle is opaque to the engine — full-path encoding is a
legibility/determinism *choice*, valid only because URL paths permit it. DB encodes what
it safely can (the slug) and hashes the rest. Both are "a deterministic
derivation of the locator into a valid handle"; only the lossiness differs.
Neither is ever decoded back to recover the target.

## Invariants

- Charset `^[a-z0-9][a-z0-9_-]*$`; `__` is reserved as the level delimiter.
- `filename == endpoint_id`.
- Unique within the owner (connector for API, connection for DB).
- The verbatim locator field is the **sole** source for building the query /
  request; the handle is never decoded back to a target.

## Structural asymmetry (by design)

API endpoints are authored by this plugin at build time; database endpoints are
generated by the engine's `resource_discovery` at runtime (connection-scoped).
Alignment is on the *principle and derivation shape*, never on who produces the
id or when.

## Immutability

An `endpoint_id` is permanent once released. Streams pin endpoints by id, so
renaming one silently detaches every stream that referenced it — the pin does
not fail loudly, it resolves to nothing. Because the handle is *derived*, this
also means a non-deterministic derivation is a correctness bug, not a style
issue: the same locator must yield the same handle on every re-author. If a
resource's locator genuinely changes, that is a new endpoint plus a removal (a
major bump), never an in-place rename.
